## 1. Architecture

```
GenerateSignature(data, blockSize) → GenerateSignatureReader(io.Reader, ...)
│
├─ Checksum1 (weak) ──────────────────────────────────────────
│   ├─ [amd64] AVX2 64B/iter  → rolling_amd64.s
│   ├─ [amd64] SSE2 32B/iter  → rolling_sse2_amd64.s
│   └─ [!amd64] byte loop     → rolling_generic.go
│
└─ Strong hash (MD5) ─────────────────────────────────────────
    ├─ Phase 1: N full 64B chunks, batched SIMD
    │     load + gather → md5x8core (or md5x16core)
    ├─ Phase 2: tail + padding (md5Finalize8way / scalar fallback)
    └─ [!amd64] crypto/md5 stub

Path selection: AVX-512 (blockSize ≥ 2KB) → AVX2 → scalar
```

## 3. Data Flow

### 3.1 Batch Buffer

Blocks are packed contiguously in a batch buffer (8× or 16× blockSize). AVX-512 activates when `blockSize ≥ 2048`; smaller blocks use AVX2. The gather overhead dominates for small blocks.

### 3.2 Gather → Transpose → Core

Each 64B chunk from 8 (or 16) blocks is transposed to Structure-of-Arrays layout: word `w` becomes a YMM (or ZMM) holding `w` from all lanes. The core reads X[g] as a single aligned `VMOVDQU`.

```
Phase 1 (batched SIMD):
  for each 64B chunk:
    gather  →  md5xNcore  →  accumulate state

Phase 2 (per-lane finalization):
  for each lane:
    process remaining full chunks (scalar)
    tail + MD5 padding  →  final digest
```

### 3.3 Implicit Register Rotation

Y0-Y3 hold (a,b,c,d), but logical roles permute each step via a compile-time table. Only one write per step: `VPADDD rb, temp, ra`. AVX-512 uses the same pattern with ZMM registers.

### 3.4 Merged T-Constant Loads

`VPADDD T_i<>(SB), Y12, Y12` (memory-source add) replaces `VMOVDQU + VPADDD`. Constants stored as 8× or 16× expanded DATA tables.

### 3.5 Software Pipelining

X[g_{i+1}] loaded at the *end* of step `i` (overlapping with writeback), not at the start on the critical path. Large gain on shallow OoO window CPUs (+67% on Zen 4); negligible on deep-window Xeons.

### 3.6 Save-Add

MD5 requires `digest += initial` after each block. Initial state saved to Y4-Y7 (Z4-Z7) on entry, added back before storing.

## 4. 8-way AVX2 Core

### 4.1 Round Functions

Three non-commutative rounds use VPANDN; Round 3 uses VPXOR (commutative).

| Round | F(b,c,d) | Implementation |
|-------|----------|----------------|
| R1 (0–15) | `(b&c) \| (~b&d)` | `VPAND rb,rc; VPANDN rd,rb; VPOR` |
| R2 (16–31) | `(b&d) \| (c&~d)` | `VPAND rb,rd; VPANDN rc,rd; VPOR` |
| R3 (32–47) | `b ^ c ^ d` | `VPXOR rb,rc; VPXOR rd` |
| R4 (48–63) | `c ^ (b \| ~d)` | `VPANDN rd,rb; VPXOR allones; VPXOR rc` |

### 4.2 Rotation

AVX2 has no single-instruction rotate. Uses `VPSLLD` + `VPSRLD` + `VPOR` (3 instructions).

### 4.3 Code Generator

`gen_md5x8/main.go` emits 64 unrolled steps with implicit register rotation, merged T-constant loads, and software-pipelined X prefetch. Output: `md5x8_amd64.s`.

## 5. 16-way AVX-512 Core

### 5.1 Key Differences from AVX2

| Aspect | AVX2 | AVX-512 |
|--------|------|---------|
| Register width | 256-bit (YMM) | 512-bit (ZMM) |
| Lanes | 8 | 16 |
| F function | VPANDN + VPAND + VPOR (3 insns) | VPTERNLOGD (1 insn) |
| Rotation | VPSLLD + VPSRLD + VPOR (3 insns) | VPROLD (1 insn) |
| Gather mask | YMM (`VPCMPEQD`) | k-mask (`KXNORW K1,K1,K1`) |
| Theoretical throughput | 1× | ~3× (2× lanes × ~1.5× IPC) |

### 5.2 VPTERNLOGD Round Functions

All F functions computed via a single VPTERNLOGD with truth-table immediate:

| Round | imm8 | Truth table |
|-------|------|-------------|
| R1 | $0xD8 | `(b&c) \| (~b&d)` |
| R2 | $0xAC | `(b&d) \| (c&~d)` |
| R3 | — | `b ^ c ^ d` (VPXOR, no VPTERNLOGD) |
| R4 | $0x63 | `c ^ (b \| ~d)` |

> **Note**: These values are for Go Plan 9 operand order. See §7.2.

### 5.3 Code Generator

`gen_md5x16/main.go` emits 64 unrolled steps with ZMM registers, VPTERNLOGD F functions, VPROLD rotation, and 64-byte T-constant tables (16× uint32). Output: `md5x16_amd64.s`.

## 6. Gather Load

### 6.1 VPGATHERDD

Collects dwords from scattered positions in one instruction. AVX2 uses a YMM mask register; AVX-512 uses a k-mask.

### 6.2 Mask Zeroing

VPGATHERDD **zeros the mask register** after execution (Intel specification). Must reload the mask before every gather:

- **AVX2**: Precompute all-ones in spare register (`VPCMPEQD Y3,Y3,Y3`), then `VMOVDQA Y3,Y2` before each gather.
- **AVX-512**: `KXNORW K1,K1,K1` before each gather.

Without reload, subsequent gathers silently read zero data.

### 6.3 VPGATHERDD Go Assembly Syntax

VPGATHERDD is supported with VSIB for both VEX (AVX2) and EVEX (AVX-512)
encodings.  The AVX2 form uses a YMM mask register as the first operand;
the AVX-512 form uses a k-mask.

| Path | Go Plan 9 syntax |
|------|-----------------|
| AVX2 | `VPGATHERDD Y2, 4(R8)(Y7*2), Y1`  *(mask, mem-vsib, dst)* |
| AVX-512 | `VPGATHERDD (R8)(Z20*1), K1, Z10`  *(mem-vsib, kmask, dst)* |

## 7. Assembly Notes

### 7.1 VPANDN Operand Order (Go Plan 9 Convention)

Go Plan 9 `VPANDN A,B,C` = `C = A &^ B` (unlike Intel's `C = ~A & B`).
This is by design in Go's assembler — it applies the same src1/src2 swap
convention to all non-commutative SIMD instructions. The AVX2 code
generator uses the Go operand order for R1, R2, and R4.

### 7.2 VPTERNLOGD Operand Order (Go Plan 9 Convention)

Same convention as VPANDN: Go Plan 9 `VPTERNLOGD imm,src1,src2,dst`
computes the truth-table index as `n = (dst<<2)|(src2<<1)|src1`,
while Intel uses `n = (dst<<2)|(src1<<1)|src2`.

The AVX-512 code generator uses Go-swapped immediates (R1=$0xD8, R2=$0xAC,
R4=$0x63). Using Intel-manual immediate values will produce wrong MD5 hashes.

### 7.3 VPMADDUBSW Operand Order (Go Plan 9 Convention)

Go Plan 9: signed operand first, unsigned operand second.
Intel manual: unsigned first, signed second.
Consistent with Go's general approach of swapping src1/src2 relative to
Intel convention.

### 7.4 VPGATHERDD Mask Initialization

Must explicitly set to all-ones. YMM: `VPCMPEQD Y,Y,Y`; k-mask: `KXNORW K1,K1,K1`. Uninitialized masks silently fill zeros.

### 7.5 Miscellaneous

- **Editing `.s` files**: Use a proper text editor, not PowerShell file redirection which may change encoding.
- **Cross-compilation**: `$env:GOOS` persists in PowerShell. Clear with `$env:GOOS=""` after cross-compiling.

- [ ] `go vet .` — zero warnings
- [ ] `go test -count=1 .` — all tests pass (including AVX-512 parity if CPU supports)
- [ ] `VZEROUPPER` before every `RET` in YMM/ZMM functions
- [ ] VPGATHERDD mask reloaded before **every** gather
- [ ] VPTERNLOGD immediates use Go-swapped operand order (not Intel manual values)
- [ ] `//go:noescape` on all asm declarations with pointer args
- [ ] Slice args occupy 24 bytes in frame (ptr + len + cap), not 16
